### 2. Deleted Events Still Visible After Calendar Disconnect ✓

**Problem**:
When disconnecting a Google Calendar account and choosing to delete events, the backend correctly deleted events from Firestore, but they remained visible in the calendar UI. The events persisted in the React component state even after navigating away and back.

**Root Cause**:
The `useCalendarEvents` hook loaded events once on mount and stored them in state, but had no mechanism to detect when events were deleted from Firestore. The hook relied solely on dependency changes (`userId`, `dayKeys`) to reload, which didn't change when events were deleted by the disconnect operation.

**Solution**: Added real-time Firestore subscription to `useCalendarEvents` hook

**File**: [apps/web-vite/src/hooks/useCalendarEvents.ts](../apps/web-vite/src/hooks/useCalendarEvents.ts)

**Changes**:

1. Added imports for Firestore real-time listeners:

   ```typescript
   import { collection, query, onSnapshot, type Unsubscribe } from 'firebase/firestore'
   import { getFirestoreClient as getDb } from '@/lib/firestoreClient'
   ```

2. Added `unsubscribeRef` to track the active listener:

   ```typescript
   const unsubscribeRef = useRef<Unsubscribe | null>(null)
   ```

3. Implemented real-time listener effect (lines 62-119):
   - Sets up `onSnapshot` listener on the user's events collection
   - Detects document changes (added, removed, modified)
   - Automatically reloads events when deletions or additions are detected
   - Properly cleans up subscription on unmount
   - Uses `active` flag to prevent race conditions

**Benefits**:

- Events automatically disappear from UI when deleted (from any source)
- Works for calendar disconnect, manual deletion, sync operations, etc.
- Proper cleanup prevents memory leaks
- Race condition protection with `active` flag

**Technical Details**:

```typescript
const unsubscribe = onSnapshot(
  q,
  (snapshot) => {
    if (!active) return

    const changes = snapshot.docChanges()
    const hasRelevantChanges = changes.some(
      (change) => change.type === 'removed' || change.type === 'added'
    )

    if (hasRelevantChanges) {
      logger.info('Events collection changed, reloading', {
        added: changes.filter((c) => c.type === 'added').length,
        removed: changes.filter((c) => c.type === 'removed').length,
        modified: changes.filter((c) => c.type === 'modified').length,
      })
      void loadEvents()
    }
  },
  (error) => {
    logger.error('Events listener error', error)
  }
)
```

## Performance Considerations

**Real-time Listener Impact**:

- The `onSnapshot` listener subscribes to the entire events collection for the user
- This is necessary to detect deletions (can't filter deleted docs)
- For users with many events (1000+), this could increase bandwidth
- Consider future optimization: filter by date range if performance issues arise
- Current implementation only reloads on add/remove changes (not on every modification)

**Batch Operation Chunking**:

- Reduced batch size from 500 to 100 operations to be more conservative
- Prevents hitting Firestore's 10 MB transaction limit
- Small performance overhead from additional commits, but ensures reliability
- Tradeoff: more network round-trips vs. avoiding catastrophic failures
